You are a new programmer and you are stuck.

This web page is a bigginers guide to debbuging. Try these suggestions and see if you can get your project working.

These suggestions are independent of language. If you ask for help too early it is hard to learn and gain confidence. If you wait too long you waste time and energy. They are steps to take before you ask for help. Each section ends with "Ask for help." As long as you are making progress keep trying.

Sometime it is hard to recognize progress.

    Signs of progress are:

  • Limiting where an error could be coming from.

    This is progress because when you find what part of your code generates the errors it is much easery to fix them.

  • Knowing what input causes errors or if it is intermittent problem.
  • Getting new error messages.

    is a sign of progess because usauly means that you have fixed one error and found a new one.

Flow Chart For Complete Newbies on what to do when Code Doesn’t work

When you get a new error message or you learn something about what or where your code is going worng, celebrate. Follow that lead. When you get stuck again start at the top of the top of this page. If you reach the end of a list of sugesstions without makeing progress thats the time to aks for help.

    Which of these best describes your problem?

    • My Code Won’t Run / I got an error or other message - (usually Syntax errors or Compile Time errors):

      Syntax errors are the best kind of error to have.

      They always come with error messages. Error messages are kindly bud dumb messages telling you where your code is confusing to the computer. (about half the time the computer is wrong about what should be fix)
      1. Read the error message: look for Red to find the error messages:
      • On what line in your program did the error occor
      • What the computer wants / key words
    2. Check Syntax, number/type/order of arguments, wrong type of function for type/array (in weakly typed goes other place)
    3. Look for an error before your line? Closing braces ";" initialization…
    4. Be your computer. check the spelling of variable names and their vlaues (sometimes your variables don't have what you think they do)
    5. Google the language + key words + sometimes function name (try variations)
    6. Comment out your line
    7. Ask for Help (explaining helps)

    • My Code runs and crashes or runs forever – (usually Run-time error):

      Don't change things randomly it causes distractions / creating other error / makes version control hard

    1. Add logs of output, check if program is making progress, where is it taking time
    2. Proof read
    3. Be your computer.
    4. Trace the decision tree. Look for flow control statements (loop, if …, switch) Check to see what parts of you code are running
    5. look for loops or requests
    6. Check your program line by line
    7. Comment out blocks of code
    8. is there anything you can Google
    9. Ask for Help

    • My Code runs but I don’t understand what it is doing or it does nothing – (usually logic errors):

      Don't change things randomly it causes distractions / creating other error / version control

    1. Look for what line/s in your code created the output you see
    2. Be your computer.
    3. Add logs/ break point in your debugger
    4. Comment out blocks of code
    5. Ask why is the computer putting this out?
    6. Look for flow control statements (loop, if …, switch) Check to see what parts of you code are running
    7. Retype / proofread your code
    8. Ask for help

    • I don't even no where to begin with this algorithm, coding problem, or code challenge:

      1. Reread instructions
      • What input/data do you get?
      • What output are you supposed to return?
      • What are you supposed to do with the data?
      2. If you are given tests or partial code:
      • Read the test: What are they testing for?
      • Try running it: What error messeges do you get?
    3. Try to write a test.
    4. White board it, write pseudo code, or explain the problem to your cat.
    5. Try to get an error message. Put your pesudo code into your code and start fixing error messeges
    6. Try this basic code pattern.
    7. Ask For Help

    • Steps for whit board exercise

    1. Question: What are you being asked to do? With what input? What is the output? What are the edge cases? What assumptions are being made?
    2. Algorithm: What process will you use to solve the problem?
    3. Complexity: How much space will it take? How much time?
    4. Test cases: Write some test input.
    5. Code: Write your code.
    6. Try your test cases in your code.
    7. Proof (optional)

    • I don't even no where to begin learning about this program, software or tool:

    1. For Code challenges
    2. if you don't understand what you are supposed to do run and see what the test are.
    3. Install make sure it is on you computer
    4. if exe try running it (usually not helpful but worth a try)
    5. YouTube name of program or tool and your operating system
    6. Tutorials / examples
    7. Google name of program or tool and your operating system
    8. Try to get an error message
    9. Ask For Help

    • Problems that go unnoticed

      1. Security – a risk when you store private data or interact with users
      • Malicious user input (when ever you take data from users, corrupt user data)
      • Transmission and storage of private data
      • Authentication
      • Lost access when passwords are lost
    2. Memory Leaks
      3. Overwrite or Corrupt Data
      • When you copy or store data
      • Over run of arrays in some languages
    4. Scaling
    5. Slow or inefficient code
    6. Bad UX and bad users
    7. Too big, small, null, wrong type, etc…
    8. Landmines how code brakes
    9. functions should work correctly and brake spectacularly

    • Suggestions by type of statement

      1. If then
      • Never nest if-statements more than two deep and always try to do them one deep.
      • Put blank lines before and after.
      • Your boolean tests should be simple. If they are complex, move their calculations to variables earlier in your function and use a good name for the variable.
      • Watch for all or nothing cases not covered. In "complete" lists. Practice for getting user input. Use of laundry/grocery lists vs example lists;
    2. Conditional
    3. For loops
    4. Do while loops
    5. While loops
    6. Not while loop or Unless loop
    7. try blocks
    8. functions and methods
    9. timing

    • Suggestions for style and structure

    1. where and when to comment
    2. "Make sure parts of your programs work as you work on them. Do not write massive files of code before you try to run them. Code a little, run a little, fix a little."
    3. Code openings and closings together. "" () begin end {} '' || [] For next/end

    • Guideline for writing a tutorial

      As much as possible teach good practice for coding the way it should be done for real projects. Teach the skills they will need and use. Don't just teach the language.

    1. Do a real project.
    2. Write test first.
    3. Start from non working code.
    4. Teach debugging starting form the errors. Show people what errors they can get and what it means.
    5. Build up code in small bits.
    6. When a section is complete refactor.
    7. Don't got strait for the best answer. Include some false starts.
    8. Include practice examples.
    9. Include different explanations for different ways of learning and thinking.
    10. Iterate your tutorail often and test your results empirically.
